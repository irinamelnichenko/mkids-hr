## 🎯 БІЗНЕС-ЛОГІКА (ФІНАЛЬНА)

### Визначення типу договору:

```
ЯКЩО дата договору >= 01.10.2025:
    → тип "sick-only" (або "new")
    → Доступний ТІЛЬКИ Лікарняний (4 тижні/рік)
ІНАКШЕ (договір до 01.10.2025):
    ЯКЩО група містить "Preschool" або "розумник":
        → тип "preschool"
        → НІЧОГО не доступно (перерахунки не передбачені)
    ІНАКШЕ:
        → тип "standard"
        → Доступна ТІЛЬКИ Відпустка (4 тижні/рік)
```

### Ліміти і знижки:

**Відпустка:**
- 1-5 роб.дн = 1 тиждень = **-25%**
- 6-10 роб.дн = 2 тижні = **-50%**
- 11-15 роб.дн = 3 тижні = **-75%**
- 16-20 роб.дн = 4 тижні = **-100%**
- Ліміт: **20 роб.дн на рік** (від дати договору + 1 рік)

**Лікарняний:**
- 1-5 роб.дн = **-5%**
- 6-10 роб.дн = **-10%**
- 11-15 роб.дн = **-15%**
- 16-20 роб.дн = **-20%** (максимум)
- 21+ роб.дн = максимум -20% (знижка тільки за перші 20)
- Ліміт: **20 роб.дн на рік**

### Загальні правила:
- Робочі дні = Пн-Пт (вихідні не рахуються)
- CRM авто-рахує робочі дні між датами через `countWorkDays(from, to)`
- Перерахунок розбивається по місяцях через `calcAbsMonthBreakdown(type, from, to, fee)`
- Знижка рахується від СУМИ ДОГОВОРУ (`monthlyFee`), не від бюджету

### Workflow статусів:
1. `pending` — директор додав, чекає HR
2. `processing` — HR в процесі обробки (опціонально)
3. `done` — HR підтвердив "✅ Внесено" → застосовується до знижки
4. `rejected` — HR відхилив
5. (новий) `cancelled` — HR скасував внесений раніше запис

### Права доступу:
- **HR** (`isHR()=true`): бачить усі записи, може робити все
- **Директор** (`S.user.loc`): може додавати/видаляти тільки свої `pending`
- **Директор** може редагувати тільки свої `pending` записи своєї локації

## 🔴 ВІДКРИТІ ЗАДАЧІ (наступні сесії)

### Задача 1: Редагування і скасування відпустки (UI + логіка)

**Правила (підтверджено 20.04.2026):**
- HR може редагувати/скасовувати **будь-які** записи (pending, processing, done)
- Директор — тільки свої **pending**
- Нотатка-причина **обов'язкова** при редагуванні
- Історія змін зберігається в `statusHistory`
- Якщо запис `done` → скасування = статус `cancelled` (не видалення)
- `recalcDiscount` ВИКЛЮЧАЄ `cancelled` (разом з `rejected`)

**Що треба зробити в коді:**

1. У `renderAbsEntry` — додати кнопки для HR на статус `done`:
   - ✏️ **Редагувати** → виклик `editAbsencePeriod(i)`
   - 🚫 **Скасувати** → виклик `cancelAbsencePeriod(i)`

2. Нова функція `editAbsencePeriod(i)`:
   - Відкрити `renderAbsAddForm` в режимі edit
   - Заповнити поля з існуючого запису
   - Обов'язкова нотатка при save
   - При save — оновити існуючий запис (не створювати новий)
   - Додати подію в `statusHistory`: `{event:'edited', at, by, from, to, reason}`

3. Нова функція `cancelAbsencePeriod(i)`:
   - Діалог підтвердження + обов'язкове поле причини
   - Статус → `cancelled`
   - Додати подію в `statusHistory`: `{event:'cancelled', at, by, reason}`
   - Викликати `writeAbsenceToPayment` (див. задача 2) для стирання в Payment

4. У `recalcDiscount` — додати фільтр `cancelled`:
   ```javascript
   var abs = (c.absences||[]).filter(a =>
      !a.id || (a.status!=='rejected' && a.status!=='cancelled')
   );
   ```

5. У `renderAbsEntry` — стилі для `cancelled` (сірий бейдж "🚫 Скасовано")

### Задача 3: Імпорт історичних даних з Payment → CRM (разова міграція)

**Масштаб:** 300+ дітей вже мають записи в BL-BO Payment файлів 17 локацій.

**Проблема:** CRM не знає про ці відпустки → HR може додати відпустку дитині у якої ліміт вже вичерпано → фінансовий ризик.

**План:**

```javascript
function importPaymentAbsencesToCRM() {
   // 1. Для кожної локації:
   //    - Відкрити Payment файл
   //    - Через detectAbsenceCols знайти колонки "1 тиждень".."4 тиждень"
   //    - Для кожного рядка дитини:
   //      - Прочитати значення BL, BM, BN, BO
   //      - Розпарсити через parseAbsencePeriod
   //      - Створити запис absence:
   //        {
   //          id: 'legacy_'+Date.now()+'_'+i,
   //          type: 'vacation',
   //          from: '2025-11-03', // або з парсингу
   //          to: '2025-11-09',
   //          workDays: 5,
   //          weeks: 1,
   //          status: 'done',
   //          isLegacy: true,
   //          createdBy: 'Payment Import',
   //          createdAt: new Date().toISOString(),
   //          note: 'Імпортовано з Payment файлу',
   //          originalFormat: '11/25' // що було в клітинці
   //        }
   //    - Додати до існуючих absences (не дублювати)
   //    - Зберегти в CRM-лист
}
```

**Формати які треба підтримати:**
- `11/25` → місяць/рік без точних дат (ставимо припущення: "якийсь тиждень у листопаді 2025, 5 роб.дн")
- `20-26.04.26` → точні дати (парситься напряму)
- `20.04-03.05.26` → період через 2 місяці
- Можливо інші — треба побачити реальні варіанти з файлів

**Ризики міграції:**
- Дублі (якщо запускати двічі)
